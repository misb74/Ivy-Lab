## Why we're evaluating this

The consulting practice has a compounding problem: every engagement generates case studies, methodologies, frameworks-that-worked, frameworks-that-didn't, and client-type patterns. None of it is searchable today. It lives across Notes, scratch files, prior decks, and memory.

Knowledge-agent-template ([Vercel Labs](https://github.com/vercel-labs/knowledge-agent-template)) is a clone-and-deploy template for *Moray's consulting brain*: an AI agent that grep/finds/cats across a knowledge base of past work, deterministic and explainable, no embeddings, no vector DB. Adopting it would shift Ivy from "smart per conversation" to "smarter per engagement."

This is the bigger structural bet in the consulting-stack adoption — Hermes adds verbs, this adds bones.

## Inspection findings (without running the app)

### Architecture

- **Stack:** Next.js + Bun + Better-Auth + GitHub OAuth + Vercel AI Gateway + Vercel Blob + SQLite via NuxtHub.
- **Search approach:** No embeddings, no vector DB. Agents run `grep`, `find`, `cat` inside isolated sandboxes against connected sources. Results are deterministic and explainable.
- **Sources:** GitHub repos, YouTube transcripts, and a "custom API" plugin model. The interesting question for consulting: can the source plugin take a local-disk directory of markdown files (case studies, methodology notes)?
- **Bots:** Web chat + GitHub Issues + Discord adapters out of the box. Slack and Linear flagged as "soon."
- **Admin agent:** Built-in `query_stats`, `query_errors`, `run_sql`, `chart` for operational questions about the app itself.
- **Complexity router:** Classifies incoming questions trivial → complex, routes to cheap/expensive models accordingly. Cost optimization automatic.
- **MCP gap:** Lab is MCP-heavy (~64 servers). KAT uses Vercel AI SDK directly with `bash`/`bash_batch` tools — *not* MCP. Bridging would mean either:
  - Standing up KAT alongside Lab and talking to it via HTTP from a pp-tool wrapper, OR
  - Forking and adapting the agent-loop pattern + sandbox approach into a Lab-native MCP server.

### Environment variables required

From `apps/app/.env.example`:

| Var | Status | How to get |
|---|---|---|
| `BETTER_AUTH_SECRET` | Required | `openssl rand -hex 32` (deterministic; no external dependency) |
| `GITHUB_CLIENT_ID` | Required | Create OAuth app at github.com/settings/applications/new |
| `GITHUB_CLIENT_SECRET` | Required | Generated by step above |
| `AI_GATEWAY_API_KEY` | Optional local | Vercel dashboard (deploys use OIDC automatically) |
| `NUXT_GITHUB_SNAPSHOT_REPO` | Optional | For snapshot persistence — owner/repo format |
| `NUXT_GITHUB_TOKEN` | Optional | PAT override if not using GitHub App |
| `NUXT_DISCORD_*` | Optional | Only if shipping the Discord adapter |

**Blocker:** the GitHub OAuth client cannot be created without Moray's GitHub account credentials. This is the gate to running the acceptance query.

## Acceptance criteria (for when keys land)

The evaluation question: **can KAT surface relevant prior consulting work in response to a real query?**

Specific test:

1. Seed `~/code/scratch/consulting-kb-sample/` with 3-5 anonymized markdown files representing prior work — e.g.:
   - `finance-transformation-case-X.md` (a case study)
   - `methodology-skills-genome-mapping.md` (a methodology note)
   - `client-type-patterns-public-sector.md` (a pattern observation)
2. Point KAT at this directory as a source.
3. Issue query: *"What have I previously worked on involving finance transformation?"*
4. Confirm KAT surfaces the relevant markdown content WITH citations (which file, which line range).
5. Bonus: ask a synthesis query — *"What's a pattern I keep seeing in public sector engagements?"* — and verify KAT cross-references multiple sources rather than returning a single doc.

## Decision space (to be resolved post-acceptance)

Three options once acceptance runs:

### Option A — Adopt as a separate service alongside Lab
- Run KAT locally on `localhost:3001` (port-shifted to avoid Lab clashes).
- Build a `pp-tools/kat-pp-cli/` wrapper that hits KAT's HTTP surface.
- Talk to it from Ivy via the wrapper.
- **Pros:** zero invasive change to Lab; KAT stays up-to-date with upstream via `git pull`.
- **Cons:** two services to keep running; HTTP bridge adds latency; KAT's UX is its own (not Ivy's).

### Option B — Fork and adapt the agent-loop pattern into Lab
- Extract KAT's sandbox + grep/find/cat tool pattern.
- Wrap as a Lab MCP server (`agent-knowledge-base` or similar).
- Skill describing knowledge-base sources lives in `.claude/skills/`.
- **Pros:** Lab-native; uses MCP; Ivy's existing voice/persona carries through.
- **Cons:** significant engineering lift; loses upstream improvements unless we mirror manually.

### Option C — Shelve
- Acknowledge KAT was inspirational but Lab isn't ready for a knowledge-layer commitment.
- Revisit when the consulting pipeline genuinely demands it (e.g., when prior-engagement recall becomes a bottleneck in a live engagement).

**Tentative lean (pre-acceptance):** Option A for the first month — minimal investment, fast feedback on whether the knowledge layer earns its keep. If yes → Option B. If marginal → Option C.
